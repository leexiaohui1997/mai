# Git Commit Helper

## 执行流程

### 步骤 0：检查 Git 变更状态

**执行命令：**

```bash
git status --porcelain
```

**结果分析：**

- 输出为空 → 无变更，终止并提示"无变更可提交"
- 输出非空 → 有变更，继续执行
  - 第 1 字符非空格 → 暂存区变更（M/A/D/R）
  - 第 1 字符为空格 → 仅工作区变更或无暂存
  - `??` → 未跟踪文件

**执行逻辑：**

- 有暂存文件（首字符非空格）→ 直接进入步骤 1
- 仅工作区变更（空格开头或 `??`）→ 执行 `git add .` 后进入步骤 1

### 步骤 1：获取变更内容

```bash
# 获取变更统计
git diff --cached --stat

# 获取变更详情
git diff --cached
```

**AI 分析任务：**

1. **识别变更类型**
   - 新增功能 → `feat`
   - Bug 修复 → `fix`
   - 文档变更 → `docs`
   - 代码格式 → `style`
   - 重构 → `refactor`
   - 性能优化 → `perf`
   - 测试相关 → `test`
   - 构建/依赖 → `build`
   - CI 配置 → `ci`
   - 杂项 → `chore`
   - 回滚 → `revert`

2. **推断 Scope**
   - 按文件路径识别：`apps/auth/*` → `auth`，`packages/utils/*` → `utils`
   - 根目录配置文件 → `config` 或省略
   - 多模块变更 → `core` 或省略

3. **提炼 Subject**
   - 一句话概括核心变更（≤70 字符）
   - 祈使句语气，首字母小写，末尾无标点

4. **提取 Body 要点**
   - 识别关键变更点，使用 `- xxx` 格式逐条列出
   - 每条保持简洁，避免冗长描述

### 步骤 2：生成并提交信息

**生成规范：**

1. **语言要求**
   - Header、Body、Footer 优先使用中文
   - 技术术语可使用英文

2. **格式规范**

   ```
   <type>(<scope>): <subject>

   <body>
   ```

3. **Body 格式**
   - 使用 `- xxx` 格式逐行列出变更点
   - 每条保持简洁，避免冗长描述

4. **规范验证**
   - type 小写且在枚举范围内
   - subject 非空且不以句号结尾
   - header ≤ 100 字符

5. **多模块检测**
   - 检测到不相关的多模块变更时，提示拆分提交

6. **执行提交**

   ```bash
   # 使用临时文件提交（支持多行）
   git commit -F - <<EOF
   <type>(<scope>): <subject>

   <body>
   EOF
   ```

7. **错误处理**
   - commitlint 验证失败：提示重新编辑
   - pre-commit hook 失败：建议运行 lint-staged 修复
   - 其他 git 错误：保留现场并提供恢复指引

## 示例

### 示例 1：新功能

**变更：** 新增 JWT 认证相关文件

**生成：**

```
feat(auth): 实现基于 JWT 的身份认证

- 新增登录接口和用户认证中间件
- 更新用户模型以存储密码哈希
```

### 示例 2：Bug 修复

**变更：** 修复日期格式化问题

**生成：**

```
fix(utils): 修正时区转换的日期格式化

- 统一使用 UTC 时间戳生成报告
- 修复 1970 年前日期导致错误的边界情况
```

### 示例 3：多模块变更提示

**变更：** 同时修改认证模块和数据库架构

**响应：**

```
⚠️ 检测到两个独立变更：
   1. 认证模块：添加验证码功能
   2. 数据库模块：调整表结构

建议拆分提交：
   - feat(auth): 添加验证码功能
   - refactor(database): 更新表结构

是否继续合并提交？(y/N) [推荐 N]
```

## 注意事项

1. **部分文件提交**

   ```bash
   git add path/to/file
   ```

2. **跳过预提交检查**（紧急情况）

   ```bash
   git commit --no-verify
   ```

3. **提交原则**
   - 保持原子性：一次提交只做一件事
   - 避免无意义提交：WIP、fix typo 等
   - 复杂重构分步提交便于 Review
